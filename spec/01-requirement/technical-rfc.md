# 01.2 - RFC（技术变更）

> **适用**：纯技术变更（重构、依赖升级、工具改造、性能优化）——用户不直接感知的变化。
>
> RFC = Request for Comments，征求意见稿。比 ADR 更详细（讲方案 + 备选 + 影响 + 排期），比 PRD 更聚焦技术（不讲业务目标）。

## 何时用 RFC

✅ **该写 RFC**：

- 重构核心模块
- 升级主要依赖（如 Go 版本、kratos 大版本）
- 引入新工具链 / 新框架
- 性能优化方案
- 内部协议变更
- 数据迁移方案（大批量、跨表）
- 技术债清理（涉及多文件 / 多周）

❌ **不该写 RFC**：

- 修单个 bug → issue
- 升级 patch 版本 → issue
- 用户能感知的功能 → PRD
- 单个架构决策（"用 A 还是 B"）→ 直接写 ADR

## RFC vs ADR

| 维度 | RFC | ADR |
|------|-----|-----|
| 目的 | 提案 + 收集意见 + 决策 | 记录已做的决策 |
| 包含 | 背景、方案、备选、影响、排期 | 背景、决策、备选、后果 |
| 时机 | 决策前 | 决策时 / 决策后 |
| 是否包含实施 | ✅ 有排期 / 任务 | ❌ 不讲实施 |
| 是否可废弃 | 可（"已废弃"状态） | 可（被新 ADR 替代） |

**关系**：一个大 RFC 可能产出多个 ADR；ADR 也可能不需要 RFC（小决策直接写）。

## 编号与归档

- **编号**：`RFC-<三位序号>-<简述>` 如 `RFC-001-migrate-to-slog`
- **路径**：`docs/rfc/RFC-XXX-<简述>.md`
- **模板**：[`docs/templates/technical-rfc-template.md`](../../docs/templates/technical-rfc-template.md)（创建新 RFC 时直接复制）

## 必填字段（模板已包含）

| 字段 | 必填 | 说明 |
|------|------|------|
| 元信息 | ✅ | 状态、作者、日期、关联 ADR / issue |
| 背景 | ✅ | 痛点、为什么现在做 |
| 方案 | ✅ | 具体技术方案，可含架构图 / 代码示例 |
| 备选方案 | ✅ | ≥ 2 个，说明取舍 |
| 影响范围 | ✅ | 代码 / 运行时 / 迁移成本 / 兼容性 / 回滚 |
| 风险 | ✅ | 已知风险 + 缓解措施 |
| 排期 | ✅ | 开始 + 完成 + 里程碑 |
| 验收标准 | ✅ | 可执行可验证 |
| 实施任务 | ✅ | 1-3 天颗粒度 |
| 变更记录 | 已接受后变更时填 | 详见 `lifecycle.md` |

## 状态流转

```
草稿 → 评审中 → 已接受 → 实施中 → 已完成
                  ↓                  ↓
                已废弃            已废弃
```

- **草稿**：作者初稿
- **评审中**：广泛评审，收集意见
- **已接受**：决策完成，可以开工
- **实施中**：第一个 task 开工
- **已完成**：所有验收通过
- **已废弃**：方案被否或被新 RFC 替代

## 评审流程

1. 作者把 RFC 状态置「评审中」
2. 在 PR / IM 频道公示，**至少 3 个工作日**收集意见
3. 异议在 RFC 评论区表达（不在 IM）
4. 重大异议必须由作者明确回复
5. 作者根据反馈修订
6. owner / 架构委员会决策，状态置「已接受」或「已废弃」

## 变更追溯

RFC 已接受后的修改：

- ❌ 禁止直接改正文
- ✅ 在 `## 变更记录` 表格追加
- ✅ 重大变更（方案大改 / 范围扩大）退回「评审中」重新走流程

## 与 ADR / HLD 联动

- 实施 RFC 时如果需要做关键架构决策 → 单独写 ADR，RFC 中关联
- 实施 RFC 涉及新模块设计 → 单独写 HLD，RFC 中关联

## 自查

- [ ] 选对载体（不是 PRD 不是 ADR）
- [ ] 元信息完整
- [ ] 备选方案列了 ≥ 2 个并说明取舍
- [ ] 影响范围已评估（代码 / 运行时 / 迁移成本 / 回滚）
- [ ] 风险已识别有缓解措施
- [ ] 排期合理，有验收标准
- [ ] 评审至少 3 个工作日
- [ ] 已接受后变更走 amendment
