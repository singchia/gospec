# 01.4 - Epic（战略级需求）

> **适用**：跨多个 PRD、跨季度、需要管理层介入的方向性目标。

## 何时用 Epic

✅ **该写 Epic**：

- 跨季度的战略目标（如"国际化"、"商业化升级"）
- 涉及多个团队 / 多个服务
- 需要拆成 5+ 个 PRD
- 需要管理层资源协调

❌ **不该写 Epic**：

- 单个 PRD 能覆盖的 → PRD
- 没有清晰的业务目标 → 这是技术债务，写 RFC

## Epic 与 PRD 的关系

```
EPIC-001 国际化
├── PRD-010 多语言支持
├── PRD-011 多时区处理
├── PRD-012 多币种结算
└── PRD-013 海外合规
```

Epic 是**容器**，不是 PRD 的替代。每个 Epic 必须最终拆成可独立上线的 PRD。

## 状态流转

```
提议 → 已批准 → 进行中 → 已完成
        ↓        ↓
      已取消   已取消
```

- **提议**：Sponsor 起草
- **已批准**：管理层评审通过，资源到位
- **进行中**：第一个 PRD 进入「开发中」
- **已完成**：全部 PRD 完成，业务指标已 review

## Epic 的纪律

### 防止 Epic 失控

- **拆分及时**：Epic 批准后必须 2 周内拆出至少前 3 个 PRD
- **定期 review**：每 2 周 / 每月 review 一次进度和阻塞
- **指标量化**：Epic 的目标必须有可度量的业务指标，否则降级为 RFC 集合
- **范围控制**："不包含"列表必须明确，防止范围漂移
- **可终止**：执行中发现假设错误，允许快速取消

### 反模式

- ❌ Epic 没有明确业务指标，变成"愿望清单"
- ❌ 一个 Epic 包含上百个 PRD
- ❌ Epic 拆出来的 PRD 无法独立上线
- ❌ Epic 进行了一年，从未 review
- ❌ Epic 既包含产品功能又包含纯重构（拆成 Epic + RFC 集合）

- [ ] 业务背景明确，跨季度 / 跨团队
- [ ] 目标有业务级量化指标
- [ ] 范围"包含 / 不包含"清晰
- [ ] 已拆出至少前 3 个 PRD
- [ ] 有 Sponsor 和 review 节奏
