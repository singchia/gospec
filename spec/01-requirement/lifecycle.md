# 01.5 - 生命周期：状态流转、变更追溯、质量检查、上线复盘

> **适用**：所有载体共用的流程规则。具体载体的细节去看 `issue.md` / `rfc.md` / `prd.md` / `epic.md`。

## 状态流转

### 通用状态机

```
草稿 → 评审中 → 已确认 → 开发中 → 已完成
        ↓        ↓        ↓        ↓
      已取消   已取消   已取消   复盘中（PRD/Epic）
                                    ↓
                                  已完成
```

各载体的状态名略有差异，但语义对齐：

| 阶段 | issue | RFC | PRD | Epic |
|------|-------|-----|-----|------|
| 起草 | 新建 | 草稿 | 草稿 | 提议 |
| 评审 | triaged | 评审中 | 评审中 | （管理层评审） |
| 通过 | — | 已接受 | 已确认 | 已批准 |
| 进行 | in-progress | 实施中 | 开发中 | 进行中 |
| 完结 | 关闭 | 已完成 | 已完成 | 已完成 |
| 复盘 | — | — | 复盘中 | （季度 review） |
| 终止 | wontfix / duplicate | 已废弃 | 已取消 | 已取消 |

### 状态门禁（强制）

- ❌ 状态 ≠「已确认 / 已接受」之前**禁止开发**
- ❌ 状态 ≠「已完成」之前**禁止上线**
- ❌ 状态在「评审中 / 草稿」时 Agent 必须**先向用户确认**

## 变更追溯（Amendment）

### 红线

载体进入「已确认 / 已接受」之后：

- ❌ **禁止直接修改正文**（破坏审计、误导执行）
- ✅ 所有变更追加到载体末尾的 `## 变更记录` 表
- ✅ **重大变更**必须重新评审，状态退回「评审中」

### 重大变更判定

| 变更 | 是否重大 | 处理 |
|------|---------|------|
| 改 typo / 补充细节 | ❌ 否 | 直接改 + 记一行 |
| 加 / 删验收标准 | ❌ 否 | 直接改 + 记一行 |
| 改目标 / 量化指标 | ✅ 是 | 重新评审 |
| 大幅扩大 / 缩小范围 | ✅ 是 | 重新评审 |
| 排期延后 > 1 周 | ✅ 是 | 通知相关方 + 记一行 |
| 关键技术方案变更（RFC） | ✅ 是 | 重新评审 |

### 变更记录格式

```markdown
## 变更记录

| 日期 | 变更人 | 变更内容 | 原因 |
|------|--------|---------|------|
| 2026-04-15 | @alice | 排期延后 1 周 | 上游依赖未就位 |
| 2026-04-20 | @bob | 加验收标准：兼容 IE11 | 大客户需求 |
```

## 质量检查（评审前自查）

### 通用必填项

- [ ] 元信息完整（状态、作者、日期、关联）
- [ ] 背景和目标清晰
- [ ] 目标有**可量化指标**
- [ ] 验收标准**可执行、可验证**
- [ ] 优先级和排期已确认

### PRD 额外项

- [ ] 功能需求覆盖**主流程和边界情况**
- [ ] 非功能需求已评估（性能 / 安全 / 运维 / 兼容性）
- [ ] 涉及 API 变更已描述方向
- [ ] 涉及数据变更已描述表和字段
- [ ] 任务已拆成 1-3 天颗粒度
- [ ] 涉及关键技术决策已关联 ADR
- [ ] 新模块 / 新服务已关联 HLD
- [ ] 新服务已走过"新服务上线 Checklist"

### RFC 额外项

- [ ] 备选方案列了 ≥ 2 个并说明取舍
- [ ] 影响范围已评估（代码 / 运行时 / 迁移成本 / 回滚）

### 通用风险管理

- [ ] 依赖和阻塞已识别
- [ ] 已知风险有缓解措施
- [ ] 未验证假设已列出

## 上线后复盘（Outcome Review）

### 谁需要复盘

| 载体 | 复盘要求 |
|------|---------|
| issue | 不需要（结论已沉淀到 docs） |
| RFC | 推荐：上线后回顾技术指标是否达成 |
| **PRD** | **强制**：上线后 4 周内 |
| **Epic** | **强制**：每季度 review，结束后总复盘 |

### 复盘内容

1. **实际指标**：对照原载体的"可量化指标"
2. **是否达成**：✅ 达成 / ❌ 未达成 / ⚠️ 部分达成
3. **未达成原因**：技术 / 需求理解 / 外部因素 / 假设错误
4. **经验教训**：可推广到其他需求的发现
5. **下一步**：下一轮迭代 / 停止 / 转入维护

### 复盘归档

- PRD 的复盘写在 PRD 末尾的 `## 上线后复盘` section
- 状态流转：`已完成` → `复盘中` → `已完成`
- Epic 的季度复盘写在 EPIC 末尾的 review 段
- 重大未达成的复盘可单独写为 lessons-learned 文档归档到 `docs/postmortems/`（虽然不是事故，但教训值得记）

### 累积效应

- 季度回顾时统计：
  - PRD 达成率
  - 平均偏差（实际 vs 计划工时）
  - 常见失败模式
- 用于改进需求评审流程本身

### 反模式

- ❌ 上线后从不 review，原 PRD "已完成"就锁死
- ❌ review 只看"功能是否上线"而不看"指标是否达成"
- ❌ 未达成时找借口而非分析根因
- ❌ 经验教训不归档，下个季度重复犯错

## 流程图汇总

```
[变更想法]
    ↓
[选载体: issue/RFC/PRD/Epic]  ← README.md 决策树
    ↓
[草稿] → [评审]
            ↓
        [已确认]  ← 状态门禁，开发前必须到达此状态
            ↓
   [拆 task / 起分支 / 编码 / 测试]
            ↓
        [开发中]
            ↓
        [上线]
            ↓
        [已完成]
            ↓
[复盘（PRD / Epic 强制）]  ← 上线后 4 周
            ↓
        [归档]
            ↓
[季度回顾 → 改进流程]
```
