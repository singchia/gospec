# 12 - 运维与事故响应

> SDLC 右移：代码合并 ≠ 工作结束。每次变更都是风险，每次事故都是改进的机会。

## 何时读哪个

| 当前任务 | 读这个 |
|---------|--------|
| 部署上线、变更管理、回滚、feature flag、灰度 | `deployment.md` |
| On-call 值班、告警响应、Runbook、Postmortem | `incident.md` |
| 容量评估、压测、混沌工程 | `capacity.md` |
| 备份策略、灾难恢复、RPO/RTO | `backup-dr.md` |

## 核心原则（全局）

1. **每次变更都是风险**：变更是事故头号原因，任何上线都要可回滚
2. **渐进发布**：不直接 100% 推全量，通过灰度限制爆炸半径
3. **自动化优于人工**：部署、回滚、切流可一键执行
4. **无指责文化**：事故复盘对事不对人，目标是改进系统
5. **从每次事故中学习**：每次 P0/P1 必须有 action items 并跟踪落地

## 强制约束（不可违反）

- 任何变更必须有回滚方案
- 告警规则必须配 Runbook 链接
- 高风险变更走金丝雀或 feature flag
- P0/P1 事故必须产出 blameless postmortem 和 action items
