# 12.3 - 容量管理与混沌工程

> 适用：容量评估、压测、大促准备、混沌实验。

## 容量评估触发条件

- 大促 / 活动前
- 接入新大客户
- 指标接近阈值（CPU/内存 > 70%，连接池 > 80%）
- 新增核心功能

## 容量规划步骤

1. **基线**：当前 QPS、资源使用率
2. **预估**：未来 3-6 个月增长曲线
3. **压测**：找出单实例瓶颈
4. **规划**：计算实例数、数据库规格、依赖方容量
5. **预留**：headroom 至少 30%

## 压测规范

- **环境**：独立压测环境，镜像生产拓扑
- **数据**：生产数据脱敏后导入或合成数据（详见 `11-security/privacy-audit.md`）
- **目标**：找出拐点（错误率 > 1% 或 P99 > SLO）
- **定期**：核心服务每季度压测一次

工具：`vegeta` / `k6` / `wrk` / `ghz`（gRPC）。

### 压测产物

每次压测输出 `docs/ops/loadtest-<date>.md`：

- 压测目标和场景
- 拓扑、配置、数据规模
- QPS / 延迟 / 错误率曲线
- 拐点和瓶颈
- 优化建议

## Chaos Engineering

### 成熟度阶梯

1. **L1**：开发环境手动注入故障
2. **L2**：预发环境定期演练（每月）
3. **L3**：生产环境有限度演练（金丝雀范围）
4. **L4**：生产持续运行 chaos 实验

### 常见实验

| 类型 | 示例 |
|------|------|
| 主机故障 | kill pod / 节点宕机 |
| 网络故障 | 延迟注入、丢包、断网 |
| 资源耗尽 | CPU / 内存 / 磁盘打满 |
| 依赖故障 | 数据库抖动、第三方 API 超时 |
| 时钟漂移 | NTP 不同步 |

工具：`chaos-mesh` / `litmuschaos` / `toxiproxy`。

### 演练红线

- 生产演练必须有回滚开关
- 演练前通知相关方
- 演练后记录结果到 `docs/ops/chaos-<date>.md`
